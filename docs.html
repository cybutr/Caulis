Caulis

Data format & API reference — the export/import JSON and the live backend API.

Backend

Caulis runs on a small self-hosted backend (Fastify + Postgres) behind api.caulis.czeddaru.dev. It replaced an earlier Firebase Realtime Database prototype — existing gardens were migrated over, Firebase is no longer used. This page documents the export/import JSON format and the live REST API.

Export / Import JSON

Settings → Backup → Export downloads a single JSON file describing your whole garden. Import reads the same shape, either merging (keeps current plants, re-ids any collisions) or replacing them.

Top-level object

FieldTypeNotes
versionstringApp version that wrote the file, e.g. "91".
exportedAtstringISO 8601 timestamp.
gardenKeystringHuman-readable garden key (e.g. mossy-fern-42).
locationsstring[]Known room names.
queuenumber[]Plant ids in print-queue order.
plantsPlant[]The garden. See below.

Plant object

FieldTypeNotes
idnumberStable identifier. Drives the QR code (?plant={id}) — never reuse or mutate for an existing plant.
namestringCommon name.
czechstringCzech name (optional).
latinstringScientific name.
locationstringRoom/spot.
daysnumberDays since last watered. With every drives status.
everynumberWatering interval in days.
historystring[]Local YYYY-MM-DD watering dates, newest last, capped at 60.
lightstringLight label.
care, factstringCare notes, fun fact.
watering, benchmarkstringWatering tier + benchmark text.
sunlightstring[]Raw sunlight tags from the species source.
species_idnumber\|nullPerenual species id, if identified.
imagestring\|nullPreset/species photo URL.
photosstring[]User photos as base64 data URLs (cover first). Can be large — stored separately server-side, not inline on the plant.
Status rule: r = days / every → r ≥ 1 needs water, r ≥ 0.7 water soon, else healthy.
QR safety: The QR payload is caulis://plant/{id}. Reordering never changes id. On merge import, only colliding incoming plants get fresh ids — existing plants keep their ids and printed tags.

REST API

Base URL: https://api.caulis.czeddaru.dev. Everything below is live.

Model

A garden is a row keyed by a human-readable key string with a bcrypt-hashed password — the same key + password you type into Settings. Logging in returns a JWT bearer token; every authenticated call sends Authorization: Bearer <token>. Plants, locations and the print queue sync as one JSON blob per garden; photos sync separately per plant.

Auth

POST/api/gardens/register
Body { key, password }. Creates a garden, returns { token }. 409 if the key is taken.
POST/api/gardens/login
Body { key, password }. Returns { token }. 404 unknown key, 401 wrong password.
GET/api/gardens/exists?key=
Availability check for a key, used before renaming a garden. No auth.
PATCH/api/gardens/password
Body { newPassword }. Changes the current garden's password.
DELETE/api/gardens/me
Deletes the current garden and all its data.

Garden sync

GET/api/garden
Returns { plants, locations, queue, updated_at }.
PUT/api/garden
Body { plants, locations, queue }. Full replace — the client always pushes the whole garden.

Photos

PUT/api/garden/photos/{plantId}
Body { photos: string[] } (base64 data URLs, cover first).
GET/api/garden/photos/{plantId}
Returns { photos }.
DELETE/api/garden/photos/{plantId}
Removes a plant's photo set.

API keys & proxying

Perenual, Anthropic, and other third-party API keys are stored server-side per garden, never sent to those services directly from the browser.

PUT/api/garden/keys
Body — any of { perenualKey, anthropicKey, plantIdKey, housePlantsKey }. Stores whichever are present.
GET/api/garden/keys/status
Returns which keys are set as booleans, never the values.
POST/api/ai/messages
Proxies to Anthropic's Messages API using the garden's stored key. Body is passed through as-is (model, messages, tools, …).
GET/api/perenual/search?q=
Proxies Perenual species search using the stored key.
GET/api/perenual/species/{id}
Proxies Perenual species detail lookup.

General-purpose tools

A couple of small utilities on the same box, unrelated to gardens — free to use from anything else pointed at this API. No auth; nothing here is sensitive beyond whatever's pasted in.

POST/api/tools/paste
Body { content, ttlMinutes? } (default 1 day, max 30 days, 200KB cap). Returns { code }.
GET/api/tools/paste/{code}
Returns { content, expiresAt }. 404 once expired.
POST/api/tools/shorten
Body { url }. Returns { code, shortUrl }.
GET/s/{code}
302-redirects to the original URL, counts the hit.
GET/api/tools/shorten/{code}/stats
Returns { url, hits, created_at }.

AI care review

Plant data sources (Perenual, House Plants, Wikipedia, PlantNet) are inconsistent — they often omit the fun fact or care notes, sometimes the watering interval, and occasionally return values that are botanically wrong (a cactus tagged "Frequent"). When an Anthropic API key is set in Settings → API keys, Caulis runs each resolved species record through claude-haiku-4-5 before it reaches the form, via the /api/ai/messages proxy above.

The model is told to keep plausible values and only fill blanks or fix clearly-wrong ones — succulents/cacti get a long watering interval, ferns and carnivorous plants a short one. It returns a watering interval (days), watering label, light, care notes, a fun fact, and the Czech common name when none was found from the other sources. Results are cached per species in localStorage (caulis_ai_care), so each species costs one call.

This only ever touches care fields — never a plant's id, so QR codes are unaffected. The pass is fully optional: with no key, or on any network/parse error, Caulis falls back to the raw source data and built-in heuristics.

Plants added before AI review existed can be upgraded in bulk: Settings → API keys shows a “Recheck older plants with AI” button (only when a key is set and such plants exist) that re-runs each one through the model and stamps it so it won't be re-processed.
