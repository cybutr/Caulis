Caulis

Data format & API reference — for the export/import JSON and a proposed REST API.

Prototype

Caulis is an in-browser React prototype. Data lives in localStorage and (optionally) a Firebase Realtime Database node. This page documents the export/import file format you can produce from Settings → Backup, and sketches a hypothetical REST API for a future backend.

Export / Import JSON

Settings → Backup → Export downloads a single JSON file describing your whole garden. Import reads the same shape, either merging (keeps current plants, re-ids any collisions) or replacing them.

Top-level object

FieldTypeNotes
versionstringApp version that wrote the file, e.g. "59".
exportedAtstringISO 8601 timestamp.
gardenKeystringHuman-readable garden key (e.g. mossy-fern-42).
locationsstring[]Known room names.
queuenumber[]Plant ids in print-queue order.
plantsPlant[]The garden. See below.

Plant object

FieldTypeNotes
idnumberStable identifier. Drives the QR code (?plant={id}) — never reuse or mutate for an existing plant.
namestringCommon name.
czechstringCzech name (optional).
latinstringScientific name.
locationstringRoom/spot.
daysnumberDays since last watered. With every drives status.
everynumberWatering interval in days.
historystring[]Local YYYY-MM-DD watering dates, newest last, capped at 60.
lightstringLight label.
care, factstringCare notes, fun fact.
watering, benchmarkstringWatering tier + benchmark text.
sunlightstring[]Raw sunlight tags from the species source.
species_idnumber\|nullPerenual species id, if identified.
imagestring\|nullPreset/species photo URL.
photosstring[]User photos as base64 data URLs (cover first). Can be large.
Status rule: r = days / every → r ≥ 1 needs water, r ≥ 0.7 water soon, else healthy.
QR safety: The QR payload is https://<host>/?plant={id}&g={node}. Reordering never changes id. On merge import, only colliding incoming plants get fresh ids — existing plants keep their ids and printed tags.

Proposed REST API

Not implemented. A design sketch for a future backend, mirroring the data shape above. Today Caulis talks directly to Firebase Realtime Database; this is what a thin server in front of it could expose.

Model

A garden is addressed by its storage node = SHA-256(key + "::" + password) — the same derivation the app uses, so the password gates access at the data layer. Plants are nested under a garden.

Endpoints

GET/v1/gardens/{node}
Whole garden (plants, locations, queue). 404 if the node is empty.
GET/v1/gardens/{node}/plants
List plants.
GET/v1/gardens/{node}/plants/{id}
Single plant (the resource a QR resolves to).
POST/v1/gardens/{node}/plants
Create a plant; server assigns the next free id. Returns the created plant.
PATCH/v1/gardens/{node}/plants/{id}
Partial update (location, care, every, …).
POST/v1/gardens/{node}/plants/{id}/water
Log a watering. Body { "daysAgo": 0 } → sets days and appends to history.
DELETE/v1/gardens/{node}/plants/{id}
Remove a plant.
POST/v1/gardens/{node}/import
Bulk import. Query ?mode=merge|replace, body = the export JSON above.

Auth

The node itself is the capability — knowing it (derived from key + password) grants access. A real deployment should add a bearer token and rate limiting; the password should never travel in the URL.

Example — log a watering

POST /v1/gardens/9f3a…/plants/1/water
Content-Type: application/json

{ "daysAgo": 0 }

→ 200 OK
{ "id": 1, "days": 0, "history": ["2026-05-27", "2026-06-01", "2026-06-03"] }

AI care review

Plant data sources (Perenual, House Plants, Wikipedia, PlantNet) are inconsistent — they often omit the fun fact or care notes, sometimes the watering interval, and occasionally return values that are botanically wrong (a cactus tagged "Frequent"). When an Anthropic API key is set in Settings → API keys, Caulis runs each resolved species record through claude-haiku-4-5 before it reaches the form.

The model is told to keep plausible values and only fill blanks or fix clearly-wrong ones — succulents/cacti get a long watering interval, ferns and carnivorous plants a short one. It returns a watering interval (days), watering label, light, care notes, a fun fact, and the Czech common name when none was found from the other sources. Results are cached per species in localStorage (caulis_ai_care), so each species costs one call.

This only ever touches care fields — never a plant's id, so QR codes are unaffected. The pass is fully optional: with no key, or on any network/parse error, Caulis falls back to the raw source data and built-in heuristics.

Plants added before AI review existed can be upgraded in bulk: Settings → API keys shows a “Recheck older plants with AI” button (only when a key is set and such plants exist) that re-runs each one through the model and stamps it so it won't be re-processed.

Key caveat: the prototype calls the Anthropic API directly from the browser (anthropic-dangerous-direct-browser-access), so the key lives in localStorage on the device. As with the other API keys, a production build must route this through a serverless proxy and never expose the key client-side.
